# 小红 × 鲁班 双 Agent 机制说明（对外分享版）

## 1) 这是什么
我们把 AI 助手拆成两个“有明确分工、可协同”的 Agent：

- **小红（主助手）**：偏通用协调、内容生产、日常助理
- **鲁班（专项助手）**：偏工程实现、系统运维、维修/执行型任务

核心目标：
1. **分工更清晰**（不同角色做自己擅长的事）
2. **稳定性更高**（一方异常时另一方可接管）
3. **风险更可控**（隔离上下文，降低串台与误操作）

---

## 2) 具体实现方式

### 2.1 Agent 级隔离（最关键）
每个 Agent 都有独立工作区与人格配置：

- 独立 workspace（文件、记忆、技能可按需隔离）
- 独立身份定义（IDENTITY.md / SOUL.md）
- 独立长期记忆（MEMORY.md + memory/*.md）

这意味着：
- 小红不会天然继承鲁班上下文
- 鲁班也不会天然污染小红记忆
- 需要协同时通过“显式通信”传递信息

### 2.2 模型与路由策略
两个 Agent 可以有不同默认模型、fallback 链路与调用习惯：

- 主模型可不同（如一个偏稳定对话、一个偏执行）
- fallback 不同（提高整体可用性）
- 任务按类型路由到更合适 Agent

### 2.3 会话级通信机制
跨 Agent 协作通过会话通信完成（而不是直接共享脑内上下文）：

- 用会话消息传递任务与结果
- 明确输入/输出边界（谁提问、谁产出、谁最终对用户回复）
- 出问题可审计（能追踪是哪一环出错）

### 2.4 能力沉淀到 Skill（而不是散落在记忆）
稳定可复用能力应写成 Skill（脚本 + SKILL 文档），而不是只记在某个 Agent 的临时记忆里。

好处：
- 新会话也能直接复用
- 两个 Agent 统一调用方式
- 降低“我记得/你不记得”的不一致

---

## 3) 双 Agent 的核心好处

### 3.1 互救（故障接管）
当一个 Agent 出现异常时，另一个 Agent 可快速接管：

- A 端模型抖动/超时 → B 端继续服务
- A 端上下文污染 → B 端用干净上下文接手
- A 端工具链异常 → B 端走备用路径

这是双 Agent 机制最实用的价值之一：**不中断服务**。

### 3.2 降低串台/幻觉风险
把“上下文隔离 + 显式通信”作为默认策略后：

- 旧任务不容易误触发
- 取消任务不容易被旧记忆反复拉起
- 错误更容易定位（知道错在哪个 Agent）

### 3.3 专业化分工
- 小红更适合内容沟通、产品/运营类输出
- 鲁班更适合工程执行、系统诊断、实操落地

同一需求可先由小红做“产品化表达”，再由鲁班做“工程化实现”。

### 3.4 可治理、可审计
- 权限边界清晰（谁能改什么）
- 变更可追溯（文件/会话级记录）
- 对外输出更稳定

---

## 4) 推荐协作规则（实践建议）

1. **所有稳定能力写 Skill，不写“口口相传”记忆**
2. **跨 Agent 默认只通信，不直接改对方私有文件**（除非明确授权）
3. **定时任务单点归属**（避免两个 Agent 同时触发同类任务）
4. **失败要降级**（主路径失败后自动走备用路径）
5. **对外只保留一个最终发言人**（减少重复和冲突）

---

## 5) 一句话总结
双 Agent 不是“两个聊天机器人并排放”，而是一个有边界、有分工、有接管能力的协作系统。  
它把“效率、稳定、可控”同时提升，尤其适合真实长期生产环境。

---

> 如果你要做团队级落地，可以继续扩展成“三层架构”：
> - 前台 Agent（用户沟通）
> - 执行 Agent（自动化实现）
> - 审计 Agent（质量与安全把关）
